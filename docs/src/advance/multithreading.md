# Multi-threading

`UnifiedSparseGrids.jl` currently uses two distinct levels of parallelism inside the unidirectional engine.
They target different bottlenecks and are selected independently of the mathematical operator definition.

## Execution knobs

The main entry point for tensor sweeps is still

```julia
apply_unidirectional!(u, grid, op, plan; parallel=:auto, parallel_width=Threads.threadpoolsize(:default))
```

For ordinary tensor sweeps, `parallel_width` controls **fiber-level** threading.
For `UpDownTensorOp`, the `parallel` keyword additionally selects the outer backend:

- `parallel=:fiber`: keep the term loop serial and thread inside each sweep over fibers,
- `parallel=:term`: parallelize over UpDown terms and keep each inner sweep serial,
- `parallel=:auto`: choose between the two from operator-aware cost estimates.

The implementation always treats `parallel_width` as a width budget for Julia's `:default` thread pool.

## 1. Fiber-level parallelism

The low-level primitive `_apply_lastdim_cycled!` applies one 1D line operator to all last-dimension fibers of the current orientation.
When the current orientation has enough work, the fibers are partitioned into chunks and processed concurrently.

Recent refactoring changed two important details:

1. Chunking is no longer based only on raw fiber length.
   The engine now uses an operator-aware proxy cost per fiber, so a dense `LineMatrixOp`, a triangular `LineBandedOp`, and a transform-like operator can produce different chunk boundaries on the same layout.
2. Chunk scratch for floating-point work buffers is stored as **one vector per worker** rather than matrix columns.
   This avoids FFTW alignment failures for transform plans, while keeping the row-pointer table contiguous.

In practice this backend works best when:

- the active orientation has many fibers,
- no single long fiber dominates the work,
- the number of UpDown terms is small.

## 2. Term-level parallelism for `UpDownTensorOp`

For an additive `L+U` split, the sparse-grid tensor product expands into several triangular terms.
The term backend assigns groups of those terms to worker tasks.
Each worker owns a private execution workspace and accumulates into its own local buffer.

Important consequences of the current design:

- inner sweeps are intentionally serial in term mode,
- task-local workspaces are used instead of indexing scratch by `threadid()`,
- the final reduction is deterministic at the worker-buffer level.

The specialized UpDown kernel now avoids several whole-vector passes that were present in the earlier refactor:

- it starts directly from the shared rotated input buffer,
- it carries buffer parity across the two phases without normalizing back between them,
- it accumulates the final sweep or final rotation directly into the worker accumulator.

That makes term mode much more attractive in higher dimensions, especially when many split terms exist but each individual sweep has weak fiber saturation.

## Shared metadata vs. mutable workspace

`CyclicLayoutPlan` is split into two parts:

- `meta`: shared orientation metadata and read-only plan caches,
- `workspace`: mutable scratch owned by one execution context.

The shared `meta` part contains items such as:

- cyclic orientation layouts,
- refinement caps,
- cached 1D line plans,
- cached chunk plans.

The `workspace` part contains mutable buffers such as:

- `write_ptr`,
- `scratch1`, `scratch2`, `scratch3`,
- `unidir_buf`,
- `x_buf`,
- `work_buf`,
- `acc_buf`.

Term workers allocate or reuse their own `workspace` objects and share only the read-only `meta` state.

## How `:auto` now decides

For `UpDownTensorOp`, automatic selection no longer uses only the number of dimensions or the number of cyclic orientations with many fibers.
Instead it estimates work from:

- the actual split mask,
- candidate omission dimensions,
- the actual term schedule generated by the two-phase UpDown execution,
- operator-aware sweep costs in each physical dimension,
- the expected fiber saturation of those sweeps.

The same estimates are also used to weight term partitions and fiber chunks.
This is particularly important on anisotropic grids where fiber lengths vary strongly by direction.

## Guidance for custom operators

If you add a new planned line operator and want it to be safely threadable:

- keep plan entries read-only during application,
- do not store mutable scratch inside the plan,
- use the caller-provided `work` buffer for all temporary memory.

This is the same contract described in [Conventions](@ref), and it is what allows both parallel backends to share cached plans safely.

## Practical advice

- Reuse `CyclicLayoutPlan(grid, T)` across repeated calls.
- Leave `parallel=:auto` unless you are benchmarking a specific problem family.
- If your workload is low-dimensional with many fibers per sweep, `:fiber` is usually the right manual override.
- If your `UpDownTensorOp` produces many terms and each term has weak per-sweep fiber saturation, try `:term`.
